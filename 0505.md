## 直接回答几个核心问题

### 1. CodeBuddy 单独用够不够?

**不够,但作为 IDE 内的副驾是必要的**。CodeBuddy 腾讯混元/DeepSeek 加持,在 IDE 里做 inline 补全、单文件理解很顺手,但 LLVM 这种规模的 rebase 有几个它**结构性不擅长**的事:

- **跨文件长链推理**(一个 conflict 涉及 IR + Pass + TableGen + Test 5-6 个文件),需要 agent loop + 自主跑 build/test,CodeBuddy 的 agent 模式在 C++ 大型 monorepo 实测不如专门 agentic 工具
- **批量 cherry-pick + 自动重试**,需要 headless 跑批,CodeBuddy 主要是 IDE-bound 交互
- **和你已有的 autoresearch / planning with files / gerrit / xwiki / trilium skills 的深度集成**,CodeBuddy 不是 skill-based 架构

**建议定位**:CodeBuddy 留给开发者**做单 commit / 单文件的精修和 review**,不做主驱动。

### 2. autoresearch 做主 agent 行不行?

**可以,但要看清它的本质**。Karpathy 的 autoresearch(我理解你指的是他那种"用 LLM 跑 research loop"的范式)本质是**一个轻量的 plan-act-observe loop**,优势是简单、可控、好 hack。劣势是:

- **没有专门为 codebase 长任务优化**(比如 token budget、subagent 隔离、checkpoint/resume)
- **没有原生的 git/build/test 工具集成**,你得自己包 MCP-like wrapper
- **多轮长 context 容易飘**,M2.5 的上下文一耗光就崩

**但是**——结合你说的"M2.5 敞开用、没 token 限制"这一条,autoresearch 反而**比 Claude Code 那套更适合你**,因为后者的杀手锏(prompt caching、Batch API、Opus/Sonnet 分层)你都用不上,M2.5 的"无限敞开"是你的最大杠杆,**就该用一个能反复跑、容忍粗暴 retry 的简单框架**。

### 3. 推荐的 Agent 选型(基于你的实际牌)

```
主驱动:autoresearch (改造版) + M2.5
├── 长 context 装 LLVM release notes + CLAUDE.md + 当前 conflict 三路 diff
├── 工具 = bash/git/grep/edit + 你的 6 个 skill
└── 多 worker 并行(每个 worker 独立 M2.5 session,反正不限量)

副驾:CodeBuddy (IDE 内)
└── 工程师手工精修、PR review、单 commit debug

兜底:Qwen Coder
└── 只做"无脑批量替换"(Batch API 风格的 mechanical change),不做语义判断
```

## 上下文是你的真正瓶颈,要专门设计

M2.5 不限 token 这条听上去爽,但**单次 context 还是受限**(你没说具体多少,但内网部署的 M2.5 通常 128K-256K)。LLVM rebase 的痛点恰好是"单个 commit 涉及的相关上下文容易爆窗口"。设计 autoresearch 的 prompt loop 时要严格遵守:

1. **Skill 检索结果必须先 distill 再喂模型**,xwiki 文档动辄几万字,直接塞进去就废了
2. **每个 sub-task 单独开 session**,不要让一个 session 跨 commit
3. **diff3 的三路 conflict 才喂全文**,周边代码用 grep 摘要
4. **build error 用 hook 只截 `error:` / `note:` 行**(我文档里写的 100× token 节省那条直接复用)

## 你应该再做的 skill 清单

你现在有 6 个 skill(planning with files、autoresearch、gerrit 历史查询、xwiki、trilium 后端优化文档、应该还有 conflict 相关)。针对 LLVM 19→22 这个具体场景,**强烈建议再加这几个**,按优先级排:

### P0(没有这些 AI 会大量翻车)

**1. `llvm-api-grounding` skill**
- 输入:任意 LLVM API 名(类、方法、attribute、intrinsic)
- 工作:在你 fork 的 LLVM 22 源码 + clangd index 里 grep 真实定义,返回签名 + 所在 header + 1-2 个 caller 示例
- 价值:**直接堵死"AI 编造 LLVM 22 不存在的 API"这个头号失败模式**。M2.5 训练数据大概率没覆盖 LLVM 21/22 的新接口,不做 grounding 必然瞎编

**2. `update-test-checks` skill**
- 包装 `update_llc_test_checks.py` / `update_test_checks.py` 等 6 个官方脚本
- 自动识别测试首行 `; NOTE: Assertions have been autogenerated` 头,决定能否机器重生成
- 输出 BEFORE/AFTER CHECK 行 diff,让 agent 决定接受还是 escalate
- 价值:**直接落地你原始问题里"是 conflict 解错还是 upstream 变化"那个判定逻辑**

**3. `alive2-verify` skill**
- 包装 `alive-tv` 二进制,输入两个 IR 函数,输出语义等价/不等价
- 任何 AI 改 IR transformation 的 patch 自动调用
- 价值:**silent miscompile 的最强防线**。你内网应该能编译 alive2,这个 skill 离线可用

**4. `git-conflict-context` skill**
- 输入:冲突文件路径
- 输出:`git show :1:file` (base) / `:2:file` (ours) / `:3:file` (theirs) 三路全文 + 双方 commit message + 双方 commit 改这个文件的最近 3 个历史
- 价值:**M2.5 看 conflict marker 的能力比 Claude 弱**,必须把三路上下文显式 concat 好喂进去,光看 marker 它会瞎选边

### P1(显著提升质量和速度)

**5. `lit-failure-triage` skill**
- 跑 `lit -vv --dump-input=always <test>`,解析输出
- 自动分类:`PURE_TEXT_DRIFT` / `SEMANTIC_REGRESSION` / `DOWNSTREAM_BREAK` / `FLAKE`
- 对前者自动调 update-test-checks skill;对后两者输出 bisect 命令
- 价值:**这就是你原始问题的核心场景,做成 skill 后 agent 决策路径固定,不依赖 M2.5 的"灵机一动"**

**6. `downstream-patch-ledger` skill**
- 维护 `DOWNSTREAM_PATCHES.md`,记录每个下游 patch 的:意图、touched files、是否已被 upstream 合入(用 git log -S 检索)、是否拟上游化、rebase 优先级
- 跨 rebase session 持久化(写到 trilium 或本地 SQLite)
- 价值:**第二次、第三次 rebase 时直接复用,投资回报最高的 skill**

**7. `tablegen-expand` skill**
- 包装 `llvm-tblgen --print-records` / 生成 `.inc` 文件
- 给 agent 一个 `.td` 文件,自动展开 multiclass/defm 后给出实际定义清单
- 价值:**`.td` 是 AI 翻车重灾区**,因为 multiclass/X-Macro 展开后才是真实代码。强制 agent 在改 `.td` 前先看展开结果

**8. `gerrit-history-deep`**(如果你现有的 gerrit skill 比较浅,加深它)
- 给定一个文件路径或符号,返回 metax 内部历史上对它做过的所有修改 + review comments
- 价值:解 conflict 时 agent 能看到"为什么 metax 当年这么改",不会盲改

### P2(锦上添花)

**9. `bisect-runner` skill**:封装 `git bisect run`,处理 build skip(exit 125)、自动 reduce(`llvm-reduce` / `creduce`)
**10. `bootstrap-diff` skill**:跑 stage1→stage2→stage3,对比二进制 hash
**11. `csmith-fuzz` skill**:跑 csmith/yarpgen 差分测试,差异自动 reduce

## 关于 conflict 自动合并的 skill,我之前文档里提到的能离线复用的

你问"还有哪些 skill"——我文档里点过的、可以本地化成 skill 的:

- **clang-include-cleaner**:LLVM 自带,堵 `#include` 删错的风险
